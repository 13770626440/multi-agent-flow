# 项目约束与流程规范

## 1. 归档要求

### 1.1 文档归档
所有项目文档必须归档到指定目录：

| 文档类型 | 归档目录 | 文件格式 | 更新频率 |
|---------|---------|---------|---------|
| 架构设计 | `docs\architecture\` | `.md` | 按需 |
| API文档 | `docs\api\` | `.md` + `.yaml` | 每周 |
| 用户文档 | `docs\user\` | `.md` | 每阶段 |
| 开发文档 | `docs\development\` | `.md` | 每周 |
| 项目计划 | `plans\` | `.md` | 每阶段 |
| 需求文档 | `requirements\` | `.md` | 按需 |
| 项目日志 | `logs\` | `.md` | 每日/每周 |
| 评审记录 | `reviews\` | `.md` + `.json` | 每次评审 |

### 1.2 代码归档
所有源代码必须归档到：
- 后端代码：`code\backend\`
- 前端代码：`code\frontend\`
- 脚本工具：`code\scripts\`

### 1.3 命名规范
- 文档：`YYYY-MM-DD-文档名称.md`
- 代码：遵循语言规范
- 评审：`YYYY-MM-DD-评审类型-版本.json`

---

## 2. 确认流程

### 2.1 文档确认流程
```
1. 撰写文档草案
2. 提交给项目负责人预览
3. 项目负责人审核并提出修改意见
4. 修改文档并重新提交
5. 项目负责人确认通过
6. 正式归档到对应目录
```

### 2.2 代码开发确认流程
```
1. 编写技术方案设计
2. 提交给项目负责人审核
3. 项目负责人批准技术方案
4. 开始代码开发
5. 完成开发后提交代码评审
6. 通过评审后合并到主分支
```

### 2.3 紧急变更流程
对于紧急修复或变更：
```
1. 立即通知项目负责人
2. 简要说明变更原因和影响
3. 获得口头或即时消息批准
4. 执行变更
5. 24小时内补全正式文档
```

## 3. 评审机制

### 3.1 评审组角色定义

#### 3.1.1 架构师（Architect）
- **关注角度**：技术架构、实现可行性、成本
- **使用模型**：GLM-5（自定义模型）
- **评审重点**：
  - 技术选型合理性
  - 系统扩展性
  - 性能与安全性
  - 技术债务控制
  - 集成复杂度

#### 3.1.2 产品经理（Product Manager）
- **关注角度**：产品需求、用户体验、商业价值
- **使用模型**：kimi2.5（自定义模型）
- **评审重点**：
  - 需求完整性
  - 用户体验设计
  - 功能价值评估
  - 用户反馈响应
  - 产品路线图对齐

#### 3.1.3 项目经理（Project Manager）
- **关注角度**：项目执行、进度控制、风险管理
- **使用模型**：DeepSeek3.1（自定义模型）
- **评审重点**：
  - 进度计划可行性
  - 资源分配合理性
  - 风险识别与应对
  - 交付物质量标准
  - 团队协作效率

### 3.2 评审标准

#### 3.2.1 评分体系（10分制）
| 分数 | 等级 | 标准 |
|------|------|------|
| 9-10 | 优秀 | 完全符合要求，无重大问题 |
| 7-8 | 良好 | 基本符合要求，有小问题 |
| 5-6 | 一般 | 需要中等程度修改 |
| 3-4 | 较差 | 需要重大修改 |
| 1-2 | 不合格 | 需要重新设计 |

#### 3.2.2 问题等级定义
| 等级 | 定义 | 处理要求 |
|------|------|----------|
| **P0** | 致命问题 | 必须修复，否则无法进入下一阶段 |
| **P1** | 严重问题 | 必须修复，影响核心功能 |
| **P2** | 一般问题 | 建议修复，影响用户体验 |
| **P3** | 轻微问题 | 可选修复，不影响核心功能 |

### 3.3 评审流程

#### 3.3.1 评审准备
```
1. 提交评审材料（至少提前24小时）
2. 材料包括：
   - 设计文档/代码/计划
   - 自评报告
   - 测试结果
   - 风险评估
3. 通知评审组成员
```

#### 3.3.2 评审执行
```
1. 架构师评审（技术角度）
2. 产品经理评审（产品角度）
3. 项目经理评审（管理角度）
4. 每个角色独立评分
5. 记录所有问题和建议
```

#### 3.3.3 评审结论
```
1. 计算平均分（三个角色）
2. 汇总所有P0/P1问题
3. 生成评审报告
4. 判断是否通过：
   - 平均分 > 9分
   - 无未解决的P0/P1问题
```

## 4. 执行要求

### 4.1 必须满足的条件
1. **评审通过**：平均分 > 9分，无未解决的P0/P1问题
2. **文档完整**：所有设计、开发、测试文档齐全
3. **代码质量**：通过代码审查，测试覆盖率达标
4. **风险可控**：已识别风险都有应对措施

### 4.2 阶段间过渡条件
从阶段N进入阶段N+1必须满足：
1. 阶段N所有交付物完成并通过评审
2. 阶段N所有P0/P1问题已解决
3. 阶段N+1计划已制定并通过评审
4. 项目负责人书面批准

### 4.3 例外处理
如遇特殊情况无法满足上述条件：
1. 必须提交例外申请
2. 说明原因和影响
3. 提供替代方案
4. 获得项目负责人特别批准

## 5. 监控与报告

### 5.1 日报要求
- 时间：每日下班前
- 内容：当日进展、问题、明日计划
- 归档：`logs\daily\YYYY-MM-DD.md`

### 5.2 周报要求
- 时间：每周五下午
- 内容：本周总结、下周计划、风险更新
- 归档：`logs\weekly\YYYY-MM-WW.md`

### 5.3 会议记录
- 所有正式会议必须有记录
- 包括：参会人员、讨论内容、决策、行动项
- 归档：`logs\meeting\YYYY-MM-DD-会议主题.md`

## 6. 变更管理

### 6.1 变更申请
任何对计划、需求、设计的变更必须：
1. 提交变更申请
2. 说明变更原因和影响
3. 评估对进度、成本、质量的影响
4. 获得评审组和项目负责人批准

### 6.2 变更记录
所有变更必须记录在：
- `logs\change-log.md`（变更日志）
- 相关文档的版本历史
- 代码的提交记录

## 7. 质量保证

### 7.1 文档质量
- 结构清晰，逻辑完整
- 语言准确，无歧义
- 格式规范，易于阅读
- 版本控制，历史可追溯

### 7.2 代码质量
- 遵循编码规范
- 有充分的注释
- 通过单元测试
- 代码审查通过

### 7.3 交付物质量
- 符合需求规格
- 通过验收测试
- 文档齐全
- 可部署、可维护
